12/3 水
Git 初回セットアップ
１：git init
２：git add .
３：git commit -m "initial commit"

[日常の作業ルール（Git運用）]
・小さな変更ごとに：git add .
・変更内容を伝える：git commit -m "変更内容を書く"

（（
  プレフィックス　頭につける拡張子らしい
  ・feature/ または feat/：新機能開発​
  ・fix/ または bugfix/：バグ修正​
  ・hotfix/：緊急修正​
  ・refactor/：コード改善（機能変更なし）​
  ・docs/：ドキュメント更新​
例（画像投稿機能：git checkout -b feature/image-upload

  ユーザー認証：git checkout -b feature/user-login

  しりとり連鎖：git checkout -b feature/shiritori-chain）
））

・ブランチで安全に機能追加：git checkout -b feature-<機能名>

・コミットツリー、mainブランチの状態や分岐を一目で把握：git log --graph --oneline --all　

・作り終えたら：git checkout main ; git merge feature-<機能名>


GitHubにログインし「New repository」をクリック

リポジトリ名（例: shiritori-app）を入力

公開or非公開設定やREADME初期化を選択

「Create repository」をクリックしリモートURLが発行される

ローカルプロジェクトで

text
git remote add origin <リモートURL>
git push -u origin main
で接続・初回push完了

次回は、macに自動で保存されるgithubトークンを交互にプロジェクト開発するために、

吉見案
リレー漫画が結構形になってきてアイデアに詰まったので友達に意見をもらった。
・同時に2人が投稿できないように任意で時間を選択させ、投稿宣言するボタンを押すと、時間制限ないは他の人は投稿できないように他の人が投稿中で、残り時間表示をする。　
・各コミックには、しっかり描きたい人が多いのか、ささっと描きたい人が多いのかが明示するために、平均待ち時間を表示
・１コミックに投稿できるコマに上限を設ける
・一応フリーに加えログイン機能をつけ、匿名だが個人にIDをつけて、その人にメッセージをおくれるようにしたり、その人は宣伝したければ自身のXリンクに自身の表示欄などに表示できる
・コミックを作成する人は、応募人数制限を設定できる

以上です。

僕としては、早く自分のコマを投稿して変なクレイジーなコミックにしたり、急に知らない人が僕の続きのコマに割り込んできて、カオスにして遊び相手が欲しい。
だから、はじめから機能もりもりではじめるべきなのかどうなのかわからない。
僕の目的は、カオスを楽しめる友達が欲しいということ。
